# XZ Update Plan - 修改版本计划的 todolist

你收到的参数: `$ARGUMENTS`

其中 `$0` 是版本号 N，`$0` 之后的内容是操作描述（修改/新增/删除要求）。

### 参数校验

如果 `$ARGUMENTS` 为空或 `$0` 不是正整数，**立即停止**，提示：

> 缺少版本号。用法: `/xz-update-plan N 操作描述`
> 示例: `/xz-update-plan 1 修改 #3 增加 refresh token 逻辑`

**示例调用：**
- `/xz-update-plan 1 修改 #3 增加 refresh token 逻辑`
- `/xz-update-plan 1 新增一条: 添加日志中间件`
- `/xz-update-plan 1 删除 #4`
- `/xz-update-plan 1 在 #2 后面插入一条缓存优化`
- `/xz-update-plan 1 测试方案改成跑 3 轮`（调整轮数）
- `/xz-update-plan 1 测试方案补一条 a < 0 的分支条件，输入 a=-1 预期报错`（增分支）
- `/xz-update-plan 1 测试方案换入口命令为 npm test`（换入口）

---

## 辅助脚本

**脚本**：`xz-tools.py`

插件启用时 `bin/` 目录自动加入 PATH，直接作为命令调用即可（不需要 `python3` 前缀，也不需要绝对路径）。脚本在**当前工作目录**下操作 `.xz_planning/`。

---

## 执行流程

### 第一步：读取项目上下文

读取 `.xz_planning/PROJECT.md`（如果存在），快速了解项目技术栈、目录结构和关键入口，为后续新增/修改条目的 change details 提供准确的文件路径参考。

### 第二步：解析当前计划

```bash
xz-tools.py parse $0
```

- 如果返回 `"ok": false` → 版本不存在，提示用户先运行 `/xz-plan N 需求描述` 创建计划
- 如果返回 `"ok": true` → 读取 N-PLAN.md 完整内容，继续

### 第三步：读取并展示当前 todolist

读取 N-PLAN.md 文件，展示当前所有条目及状态：

```
当前 todolist (版本 N):
  [x] #1 创建 User model           ← 已完成，锁定
  [x] #2 实现注册接口               ← 已完成，锁定
  [ ] #3 实现登录接口               ← 可修改
  [ ] #4 JWT 中间件鉴权             ← 可修改
  [ ] #5 添加单元测试               ← 可修改
```

### 第四步：锁定检查

**铁律：已完成的 `[x]` 条目不可修改、不可删除。**

如果用户的操作描述涉及 `[x]` 条目，**立即拒绝**并说明：

> #N 已完成，无法修改/删除。已完成的条目受锁定保护。

### 第五步：解析用户操作意图

根据操作描述，判断属于以下哪种操作（可以组合）：

#### 操作 A：修改已有条目

- 用户指定编号（如 "修改 #3"）
- 检查该编号是否为 `[ ]` 状态
- 生成修改后的条目内容（标题 + change details）

#### 操作 B：新增条目

- 在末尾追加，或在指定位置插入（如 "在 #2 后插入"）
- 插入后，后续条目编号自动递增
- 新条目必须包含完整的 change details

#### 操作 C：删除条目

- 用户指定编号（如 "删除 #4"）
- 检查该编号是否为 `[ ]` 状态
- 删除后，后续条目编号自动递减

#### 操作 D：更新测试方案

用户描述涉及测试入口 / 测试数据 / 测试环境 / 分支条件清单 / 通过标准 / 测试轮数任意一项时：

1. 定位 PLAN.md 的 `## 测试方案` 章节（如果不存在则新建）
2. 按用户描述修改对应子节（入口 / 数据 / 环境 / 分支条件清单 / 通过标准 / 轮数 / 失败后处理）
3. **新增/修改分支条件**：每条必须含**具体测试数据**（输入值 / 前置状态）；用户没明说数据时**作为「信息缺口」写进变更草案**（禁止单独发起 AskUserQuestion 问数据，合并到第八步一次确认）
4. 测试相关的 todolist 末尾条目（"运行自动化测试"）无需改编号，保持在最后
5. 测试方案若是从无到有添加（计划原本没测试），同步在 todolist 末尾插入「运行自动化测试」条目，其他条目编号不变，测试条目取新编号 N+1

### 5.1 测试必要性自动评估（写进变更草案，不单独弹问）

新增/修改条目时，AI 自动对**本次涉及的新增/修改条目**做「直观类 vs 数据逻辑类」判定（判定标准与 xz-plan 3.6 相同），然后按下表直接把建议写进第七步变更草案，禁止为此单独发起 AskUserQuestion：

| 原 PLAN 测试方案状态 | 本次条目类别 | 草案处理 |
|-----|-----|-----|
| 已有 | 数据逻辑类且新增了代码分支 | 草案里**同时列出建议追加的测试分支**（带 AI 推导的输入数据），和条目变更一并呈现 |
| 已有 | 数据逻辑类但未新增分支 | 测试方案不动，草案备注 "未新增分支，测试方案无需调整" |
| 已有 | 直观类 | 测试方案不动，草案备注 "直观类改动，不影响测试方案" |
| 无 | 数据逻辑类 | 草案里**直接起草一个最小 `## 测试方案` 章节**（沿用 xz-plan 3.6 结构），AI 能推断的入口/分支/数据都填上，不能推断的字段写 `待补充` 并一次性列在草案末尾的 "信息缺口" 段 |
| 无 | 直观类 | 不动，草案不提测试 |

**边界模糊** → 按「数据逻辑类」处理，草案标注 `AI 判定：数据逻辑类（边界模糊，倾向保守）`。

草案里的"信息缺口"段 = 用户可在第八步确认回复中顺带补齐（不强制单独提问）；未补齐则按 `待补充` 写入 PLAN.md，留给下一轮 `/xz-update-plan` 迭代。

### 第六步：方案对比（如有多种改法）

与 `/xz-plan` 同理，在生成变更草案前**必须先判断**：用户描述的修改/新增是否存在多种实现方式？

**判断条件（任一命中就出方案对比）：**

1. 修改涉及技术选型变更（换库、换协议、换数据结构）？
2. 新增条目的实现路径不止一种？
3. 修改会影响其他已有条目的 change details？
4. 存在性能 vs 复杂度 vs 兼容性的权衡？

**命中 → 出方案对比：**

```markdown
当前修改存在多种实现方式：

### 方案 A (推荐): 方案名称

**核心思路:** 一句话说清楚

**涉及改动:**
- 修改 #3: xxx → xxx
- 新增 #6: xxx

**优点:** ...
**缺点:** ...

---

### 方案 B (备选): 方案名称

**涉及改动:**
- 修改 #3: xxx → xxx
- 修改 #4: xxx → xxx（连带影响）
- 新增 #6: xxx

**优点:** ...
**缺点:** ...

---

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| 改动条目数 | 2 | 3 |
| 复杂度 | 低 | 中 |
| 对已有条目影响 | 无 | #4 需连带修改 |

👉 推荐方案 A: [理由]
```

**输出方案对比后，使用 AskUserQuestion 工具让用户选择方案，禁止自动选方案：**

- question: "选择实现方案："
- header: "方案选择"
- options: 动态根据实际方案数构建，每个选项的 label 为方案名（如 "方案 A (推荐): xxx"），description 为核心思路一句话摘要
- multiSelect: false

用户选定后，按选定方案进入下一步。如果用户选择 Other，按其输入的方案偏好处理。

**未命中 → 直接进入下一步。** 简单的增删改（如"删除 #4"、"修改 #3 的描述文字"）不需要出方案。

### 第七步：生成变更草案

**先输出到对话中，不写文件！**

展示变更对比，格式：

```
变更对比 (版本 N):

  [锁定] #1 创建 User model         ✅ 已完成，不可变更
  [锁定] #2 实现注册接口             ✅ 已完成，不可变更
  [修改] #3 实现登录接口
    旧: JWT 签发，过期 24h
    新: JWT 签发 + Refresh Token，access 过期 1h，refresh 过期 7d
  [保留] #4 JWT 中间件鉴权           （无变化）
  [删除] #5 添加单元测试             ← 将被删除
  [新增] #5 添加 token 刷新接口      ← 新增条目

重编号后最终 todolist:
  [x] #1 创建 User model
  [x] #2 实现注册接口
  [ ] #3 实现登录接口 (已修改)
  [ ] #4 JWT 中间件鉴权
  [ ] #5 添加 token 刷新接口 (新增)
```

### 第八步：等待确认

使用 AskUserQuestion 工具确认变更：

- question: "以上变更是否确认？"
- header: "确认变更"
- options:
  - label: "确认更新", description: "写入 N-PLAN.md 并刷新 STATE.md，等手动调 /xz-exec"
  - label: "确认更新并直接执行", description: "写入后立即进入 /xz-exec N 流程"
  - label: "调整测试方案建议", description: "AI 的测试必要性判定或建议分支不符合预期，重出草案"
  - label: "修改意见", description: "调整后重新输出变更草案"
- multiSelect: false

**禁止自动写入文件。必须等待明确确认。** 用户选择「修改意见」/「调整测试方案建议」或 Other 时，按其反馈调整后重新输出草案。草案有"信息缺口"时，用户可以在回复中直接补齐数据，不用再发起单独提问。

### 第九步：确认后更新文件

**获取当前时间：**

```bash
date "+%Y-%m-%d %H:%M:%S"
```

1. **更新 N-PLAN.md**：
   - 修改/新增/删除对应的 todolist 条目和 change details
   - 重新编号所有条目，保证连续（1, 2, 3, ...）
   - **同步更新「任务总结」章节** — 根据变更后的 todolist 重新生成自然语言总结，保持与条目一一对应
   - 更新「最后更新」时间
   - 在「变更记录」中追加一行：`- YYYY-MM-DD HH:mm:ss 操作描述`

2. **刷新 STATE.md**：

```bash
xz-tools.py update-state
```

**若用户选「确认更新并直接执行」**：STATE.md 刷新完成后，直接按 `/xz-exec N` 的完整流程接着执行 todolist，跳过下面第十步（由 xz-exec 自己的流程接管后续询问）。

### 第十步：输出改动结果（仅「确认更新」路径）

整理并输出：
- 修改了哪些条目
- 新增了哪些条目
- 删除了哪些条目
- 当前 todolist 总数和完成进度

然后使用 AskUserQuestion 工具让用户选择下一步操作：

- question: "计划已更新。接下来要做什么？"
- header: "下一步"
- options:
  - label: "/xz-exec N", description: "继续执行 todolist"
  - label: "/xz-update-plan N", description: "继续修改计划"
  - label: "/xz-ref N", description: "加载计划作为上下文参考"
- multiSelect: false

用户选择后，执行对应的 skill 命令。如果用户选择 Other，按其输入内容响应。

## 关键规则

1. **已完成锁定** — `[x]` 条目绝不可修改或删除，这是铁律
2. **先输出后写文件** — 变更草案必须先展示，用户确认后才写入
3. **编号自动维护** — 插入/删除后重新编号，保持 1, 2, 3... 连续
4. **change details 必填** — 新增/修改的条目必须包含完整的 change details
5. **变更记录追加** — 每次修改都在变更记录中留痕
6. **STATE.md 同步** — 更新文件后必须刷新 STATE.md 反映最新进度
7. **多种改法必须出方案** — 涉及技术选型变更、实现路径不唯一、或影响其他条目时，必须先出 A/B 方案对比，等用户选定后再生成变更草案。简单增删改不需要
8. **Bug/问题排查类变更** — 如果操作描述涉及修复 bug、排查问题、处理异常等，建议用户先用 `/xz-debug` 定位根因，再回来新增/修改 todolist 条目。纯功能调整不需要
9. **测试必要性自动评估** — 新增/修改条目时 AI 自己判定直观类/数据逻辑类并把建议（追加的分支 / 新起草的测试方案 / 信息缺口）直接写进变更草案；禁止单独发起"是否要加测试"或"分支数据是什么"的 AskUserQuestion
10. **测试数据必须具体** — 新增/修改分支条件时，每条必须有**具体测试数据**；AI 不确定数据时必须**作为"信息缺口"写进草案**，由用户在第八步确认时顺带补齐，禁止凭空造值
11. **边界模糊保守分类** — 5.1 判不准就当数据逻辑类，草案标注"AI 边界模糊倾向保守"
12. **一次确认批处理** — 变更草案是唯一的确认入口，所有 AI 判定和建议全写进草案，第八步 AskUserQuestion 的 options 覆盖所有 override 路径；禁止在草案之前单独弹问
13. **聚焦本次操作边界** — 严格按用户的操作描述范围调整，禁止顺手扩展到其他条目（例如"新增一条日志中间件"不应连带重排其他条目的细节）
