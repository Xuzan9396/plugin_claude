# XZ Plan - 创建新版本计划

你收到的参数: `$ARGUMENTS`

**参数解析：** 第一个空格前为版本号 N，之后全部为需求描述。示例 `2 支持多语言切换` → N=2。

### 参数校验

任一不满足 → **停止**并提示 `用法: /xz-plan N 需求描述`：
- `$ARGUMENTS` 为空
- 版本号不是正整数
- 需求描述为空

---

## 辅助脚本

**脚本**：`xz-tools.py`

插件启用时 `bin/` 目录自动加入 PATH，直接作为命令调用即可（不需要 `python3` 前缀，也不需要绝对路径）。脚本在**当前工作目录**下操作 `.xz_planning/`。

---

## 执行流程

### 第一步：检查初始化

当前目录无 `.xz_planning/` → **停止**，提示 `请先执行 /xz-init`。

### 第二步：检查版本存在

```bash
xz-tools.py parse N
```

| 返回 | 操作 |
|------|------|
| `"ok": false` 且 `phase` 为空 | 新建 |
| `"ok": false` 但 `phase` 存在（如已有 DISCUSS） | 新建，复用目录 |
| `"ok": true` | **停止**，提示用 `/xz-update-plan N` |

---

## 新建计划

### 1. 调研需求 + 代码现状

**必做，不可跳过：**

1. 读 `.xz_planning/PROJECT.md`（若存在）了解技术栈、结构、入口
2. 读同目录 `N-DISCUSS.md`（若存在），其结论优先作为规划输入
3. 用 Glob/Read 深入需求涉及的模块
4. 识别影响范围：新建/修改文件、依赖

**扫描排除：** `.xz_planning/`、`.gitignore` 声明项。

### 2. 范围检查

- **多个独立子系统（无依赖）** → **停止**，输出拆分建议，每个子系统走独立 `/xz-plan`
- **Bug/问题排查类**（报错、异常、"登录失败""数据不对"、用户贴了堆栈/截图）→ **停止**，提示 `该需求属于问题排查，先通过 xz-debug 定位根因`；**禁止凭猜测规划修复**

反例（不走 xz-debug）：性能优化、UX 改良、重构、新增功能。

### 3. 理解对齐 + 方案评估

**先输出「我的理解」再输出方案**：基于第 1 步调研结果，AI 必须先把对需求的理解明确写出来，然后紧接着给出方案（单方案或多方案对比），最后合并成**一次 AskUserQuestion** 让用户同时确认「理解」和「方案选择」。

**对话形态：**

```markdown
## 我的理解
- 目标：...
- 要解决的问题：...
- 期望效果：...
- 影响范围：...

## 方案
（单方案思路 或 A/B/C 对比 + 表格）
```

- **只有一种合理做法** → 简述思路，合并进 AskUserQuestion 确认
- **有多种做法** → 列 A/B/C 对比，合并进 AskUserQuestion 确认

**铁律：**
- 方案来自代码分析，禁止凭空凑数；一种做法就一条
- 多方案之间必须有实质差异（不同库/架构/改法）
- 最多 3 个，超过必须合并或淘汰
- 优缺点必须量化（如 `改 3 文件约 45 行` / `首次加载减 40KB`），禁止"性能好""扩展强"

**多方案对比格式：**

```markdown
### 方案 A (推荐): 名称
**核心思路:** 一句话
**涉及改动:** 新建/修改/依赖各一句
**优点 / 缺点:** 量化表述
**适用场景:** 一句话
```

附对比总结表（维度：改动范围/开发量/性能/可维护性/扩展性）和推荐理由。

**决策（AskUserQuestion，一次性确认理解 + 方案）：**

- question: "理解是否准确？选择哪个方案？"
- header: "理解+方案"
- options: 动态构建；多方案时每个有效方案一项（label 形如 "理解OK - 方案 A (推荐)"，description 为方案核心思路），另加 "需要修正理解"；单方案时两项 "理解准确，继续" / "需要修正理解"
- multiSelect: false

用户选 "需要修正理解" 或 Other → 按反馈更新理解后重新输出并再次确认。确认后进入 3.5。

### 3.5 选型澄清（一次性批量确认）

方案选定后，列出实现中**所有模糊的具体选型**（库、技术、协议、存储方式等），一次性展示。

**规则：**
- 只列真有选择空间的点（项目已固定的依赖不列）
- 每项 **a = 推荐 = 最佳**，括号一句话说优势
- 单项选项 ≤ 4 个
- 无模糊选型则跳过本步
- 用户确认后才进入第 4 步

**格式：**

```markdown
以下选型请确认（默认选 a，也是当前推荐）：

1. Web 框架
   a. Gin（性能高，社区最大，推荐）
   b. Echo（中间件生态好）
   c. Fiber（基于 fasthttp，最快但生态弱）

2. 日志库
   a. zap（结构化日志性能最佳，推荐）
   b. slog（标准库 1.21+，零依赖）

回复「全部确认」采用所有默认项，或指定如 `1b、2a` 覆盖。
```

### 4. 生成 todolist 草案

**先输出到对话，不写文件！**

获取时间：`date "+%Y-%m-%d %H:%M:%S"`

**需求描述规则：** `## 需求描述` 只写「我的理解（做啥）」（一段话复述目标）。用户贴过的代码/日志/链接/约束汇入 `## 用户资料与引用`，**原文保留，禁止总结重写**，无内容则整节删除。

**change details 规则：**

1. 修改文件标 `path:起始行-结束行`，括号内写 `(现: ...)` 摘要
2. 新建文件不标行号，但列出函数/类/常量名
3. 行号必须用 Read 确认，不得猜测
4. 修改超 30 行，拆成多个子区域

**粒度：** 每条 todo 2-5 分钟可完成，能独立验证，一个功能单元一条。

**格式模板：**

````markdown
# 版本 N: 中文需求名（10-25 字，与目录名同步）

> 创建时间: YYYY-MM-DD HH:mm:ss
> 最后更新: YYYY-MM-DD HH:mm:ss
> 状态: 待执行

## 需求描述

### 我的理解（做啥）
一句话或一段话复述：这次到底要做什么、解决什么问题、期望效果是什么。用户后续若纠正，更新这段。

## 用户资料与引用

> 承载需求以外的背景资料，支持 todolist 独立执行时参考。无内容则删整节。

- **引用代码 / 数据 / 日志**：用户贴的原文片段，用代码块包住**原文保留，禁止总结重写**
- **外部链接**：API 文档、设计稿、规格说明等 URL
- **业务背景 / 术语 / 约束**：专业术语、兼容要求、历史决策

## 技术方案
选定方案及理由（或单方案思路）。

## Todolist

- [ ] 1. 任务标题

  **Files:**
  - Create: `path`
  - Modify: `path:42-58`

  ```
  change details:
  新建: path
  - 导出的函数/常量/类型
  修改: path:42-58 (现: 当前内容摘要, 改后: 目标状态)
  - 具体改动点
  ```
----------------------
- [ ] 2. ...

## 总结

共 N 条，涉及 X 文件（新建 Y，修改 Z） 。
1. 任务标题 — 做什么、涉及文件、关键函数、改成什么样
2. ...

## 改动前后对比

直观展示宏观效果变化（有数据贴数据，有请求贴请求响应,有画面使用终端画出流程或者画面表示）。

改动前:
（原始现状）

改动后:
（变化后样子）

规则：基于实际代码、数据优先、不重复 todolist。

## 变更记录
- YYYY-MM-DD HH:mm:ss 初始创建
````

### 4.5 输出测试方案（仅对话，不入文件）

草案展示完毕后，**额外在对话里输出一份清晰的测试方案**，帮助用户后续自测；此部分**不写入 PLAN.md**。

**格式：**

```markdown
## 测试方案（对话参考，不入文件）

**目前情况：** 改动前的系统 / 接口 / 数据现状
**前置条件：** 开始测试前要准备的环境、数据、账号、依赖
**操作步骤：**
  1. 第一步操作（具体命令或交互）
  2. 第二步操作
  ...
**预期结果：** 每步或整体预期的输出 / 状态 / 数据变化
```

规则：
- 测试方案来自 todolist 草案推导，覆盖主要改动路径
- 命令要可直接执行；涉及敏感信息只写引用（如 `见 .env 的 DB_PASSWORD`）
- 不涉及自动化循环、轮数、分支清单等复杂结构，只给用户人工跑一遍的指引

### 5. 等待确认

**AskUserQuestion：**
- question: "以上计划是否确认？"
- options:
  - label: "确认写入", description: "只写入 N-PLAN.md，等手动调 /xz-exec"
  - label: "确认写入并直接执行", description: "写入后立即进入 /xz-exec N 流程"
  - label: "修改意见", description: "按反馈调整草案后重出"
- multiSelect: false

选「修改意见」或 Other → 按反馈调整后重出草案。

### 6. 写入文件

- 目录已存在（`/xz-discuss` 创建过） → 直接写 `N-PLAN.md`
- 不存在 → 新建 `.xz_planning/phases/N.中文名/`（`中文名` 10-25 字，带「改动对象+动作」，无空格和 `/\:*?"<>|`，禁止单用 `优化`/`重构` 这类泛化词，例如 `订单表加软删除兼容取消流程`）

写入 `N-PLAN.md`，然后：

```bash
xz-tools.py update-state
```

**若用户选「确认写入并直接执行」**：STATE.md 刷新完成后，直接按 `/xz-exec N` 的完整流程接着执行 todolist，跳过下面第 7 步（由 xz-exec 自己的流程接管后续询问）。

### 7. 输出结果（仅「确认写入」路径）

```
计划已写入:
  .xz_planning/phases/N.xxx/N-PLAN.md
```

**AskUserQuestion：**
- question: "接下来要做什么？"
- options: `/xz-exec N` / `/xz-update-plan N`

用户选定后执行对应 skill。

## 关键规则

1. **先输出后写** — 草案先展示，用户确认后才落盘
2. **change details 必带行号和当前摘要** — 新建文件列函数/类/常量
3. **零上下文假设** — 写得像给完全不了解项目的人看
4. **原子粒度** — 每条 todo 独立可验证
5. **方案来自代码分析** — 禁止凭空凑数；多方案等用户选
6. **调研先行** — 不读代码不规划
7. **多子系统拆分** — 独立子系统各自一个版本
8. **遵循现有模式，DRY/YAGNI** — 不为假想未来预留
9. **版本已存在拒绝** — 导向 `/xz-update-plan N`
10. **我的理解 + 资料原文保留** — `## 需求描述` 必须含「我的理解（做啥）」一段；用户贴的代码/日志/链接进 `## 用户资料与引用`，原文保留不得总结
11. **理解先对齐再出方案** — 方案评估阶段先讲「我的理解」，再出方案，最后合并一次 AskUserQuestion 同时确认理解和方案
12. **测试方案只对话不入文件** — PLAN.md 不含测试章节；草案展示后在对话里额外输出一份「目前情况/前置条件/操作步骤/预期结果」格式的测试方案
13. **目录名必须具体** — `N.xxx` 的 `xxx` 含改动对象+核心动作+关键差异，10-25 字；禁空格和 `/\:*?"<>|` 等特殊字符；禁止单用泛化词（优化/重构/修复 等）
14. **能推断不问** — PLAN.md、N-DISCUSS.md、代码现状能推断的字段，直接填默认值并标 `已填=X（AI 推断）`，用户可选 override
15. **聚焦本次需求边界** — 禁止把用户原始需求外的"顺手优化"写进草案
