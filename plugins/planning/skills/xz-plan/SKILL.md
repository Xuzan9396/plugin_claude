# XZ Plan - 创建新版本计划

你收到的参数: `$ARGUMENTS`

**参数解析：** 第一个空格前为版本号 N，之后全部为需求描述。示例 `2 支持多语言切换` → N=2。

### 参数校验

任一不满足 → **停止**并提示 `用法: /xz-plan N 需求描述`：
- `$ARGUMENTS` 为空
- 版本号不是正整数
- 需求描述为空

---

## 辅助脚本

**脚本**：`xz-tools.py`

插件启用时 `bin/` 目录自动加入 PATH，直接作为命令调用即可（不需要 `python3` 前缀，也不需要绝对路径）。脚本在**当前工作目录**下操作 `.xz_planning/`。

---

## 执行流程

### 第一步：检查初始化

当前目录无 `.xz_planning/` → **停止**，提示 `请先执行 /xz-init`。

### 第二步：检查版本存在

```bash
xz-tools.py parse N
```

| 返回 | 操作 |
|------|------|
| `"ok": false` 且 `phase` 为空 | 新建 |
| `"ok": false` 但 `phase` 存在（如已有 DISCUSS） | 新建，复用目录 |
| `"ok": true` | **停止**，提示用 `/xz-update-plan N` |

---

## 新建计划

### 1. 调研需求 + 代码现状

**必做，不可跳过：**

1. 读 `.xz_planning/PROJECT.md`（若存在）了解技术栈、结构、入口
2. 读同目录 `N-DISCUSS.md`（若存在），其结论优先作为规划输入
3. 用 Glob/Read 深入需求涉及的模块
4. 识别影响范围：新建/修改文件、依赖

**扫描排除：** `.xz_planning/`、`.gitignore` 声明项。

### 2. 范围检查

- **多个独立子系统（无依赖）** → **停止**，输出拆分建议，每个子系统走独立 `/xz-plan`
- **Bug/问题排查类**（报错、异常、"登录失败""数据不对"、用户贴了堆栈/截图）→ **停止**，提示 `该需求属于问题排查，先通过 xz-debug 定位根因`；**禁止凭猜测规划修复**

反例（不走 xz-debug）：性能优化、UX 改良、重构、新增功能。

### 3. 方案评估

基于第 1 步调研结果：

- **只有一种合理做法** → 简述思路，进入 3.5
- **有多种做法** → 列 A/B/C 对比（展示即可，**不单独问**，选择动作合并到 3.6 的打包文本问答里）

**铁律：**
- 方案来自代码分析，禁止凭空凑数；一种做法就一条
- 多方案之间必须有实质差异（不同库/架构/改法）
- 最多 3 个，超过必须合并或淘汰
- 优缺点必须量化（如 `改 3 文件约 45 行` / `首次加载减 40KB`），禁止"性能好""扩展强"

**多方案对比格式（仅展示，不触发工具）：**

```markdown
### 方案 A (推荐): 名称
**核心思路:** 一句话
**涉及改动:** 新建/修改/依赖各一句
**优点 / 缺点:** 量化表述
**适用场景:** 一句话
```

附对比总结表（维度：改动范围/开发量/性能/可维护性/扩展性）和推荐理由。

### 3.5 选型澄清（与方案 + 测试一起打包）

方案预选（推荐 A）后，AI 列出实现中**所有模糊的具体选型**（库、技术、协议、存储方式等）作为打包问答的一部分，不单独弹问。

**规则：**
- 只列真有选择空间的点（项目已固定的依赖不列）
- 每项 **a = 推荐 = 最佳**，括号一句话说优势
- 单项选项 ≤ 4 个
- 无模糊选型则本段不出现
- 选型澄清与方案选择、测试类别判定、测试缺口**合并在 3.6 的打包文本问答里**一次性给用户

### 3.6 测试必要性判定 + 信息收集

**前置：AI 自动判定测试必要性**（基于第 1 步调研和第 3 步方案结论，不问用户）

- **直观表现类**（全部满足才算）：
  - 改动仅涉及静态文案、CSS 样式、图标、颜色、文字大小、布局调整 / 纯文档（Markdown/注释）/ 展示性配置（默认主题、默认语言等）
  - 不涉及条件分支、状态流转、数据读写、接口输入输出字段变化
  - 不跨文件影响其他模块的数据流
- **数据逻辑类**（任一命中即算）：
  - 新增/修改 if/switch/循环/状态机
  - 改动数据模型、SQL、API 字段、接口协议
  - 改动校验、权限、计算、筛选、排序等逻辑
  - 影响持久化（DB、缓存、文件）
  - 改动并发/事务/错误处理路径
- **边界模糊** → 保守按「数据逻辑类」处理，草案标注 `AI 判定：数据逻辑类（边界模糊，倾向保守）`；**禁止为省事硬标成直观类**

---

**判定结果 = 直观表现类 → 跳过后续 6 项收集**，直接进入第 4 步生成草案。草案里：
- 不写 `## 测试方案` 章节
- 在 `## 技术方案` 之后插入 `## 测试必要性说明`，一句话：`AI 判定：直观表现类。理由：{具体理由}。本次计划不含自动化测试，仅以「改动前后对比」呈现效果。`
- `## 改动前后对比` 必填且给足细节（文案 diff / ASCII UI 图 / 字段样例等）

---

**判定结果 = 数据逻辑类 → 走 6 项收集**，但**只问真缺口**：

6 项清单（入口命令、测试数据/目标、测试环境、通过标准、测试轮数、分支条件清单+每条数据）中：
- 凡能从用户原话、需求描述、技术方案、代码现状推断的 → AI 直接填入草案并标 `已填=X（AI 推断）`
- 剩下真正缺失的项 → 合并到「第 3 步方案选择 + 3.5 选型澄清 + 本步测试缺口」的**同一次 AskUserQuestion（或同一段打包问答）**里，让用户一次回复决定

**打包问答形式（纯文本，不用 AskUserQuestion 工具）：**

方案评估、选型澄清、测试类别判定结果、测试信息缺口全部合并为**一段编号文本**发给用户，让用户自由回复：

```
请一次回复以下批量决策（未回复项采用 AI 推荐默认）：

1. 方案选择：推荐 A（理由...）。若换 B/C 请注明
2. 选型澄清：1a（Gin，推荐）/ 2a（zap，推荐）/ 3a（...）。若 override 写如 `1b 2a`
3. 测试类别：AI 判定【数据逻辑类 / 直观表现类】，理由 XXX。若想覆盖回复「加测试」或「去掉测试方案」
4. 测试缺口（仅数据逻辑类且 AI 推断不出的才列）：
   - 入口命令：？
   - 分支 3 数据：？
   - ...

回复示例：
  `全部确认`  — 按推荐全部执行
  `方案B / 选型 1b 2a / 入口=pytest xxx / 分支3数据=a=-1`  — 混合 override
```

**禁止用 AskUserQuestion 工具触发本步打包问答，必须是纯文本编号列表。** 禁止拆成多次连问；多方案对比 + 选型澄清 + 测试判定 + 测试缺口全部合并在这一段文本里。

用户回复后：
- 仍有缺口 → **合并为一次追问**剩余缺失，禁止再拆分
- 识别代码里所有分支条件（按 if/switch/状态机推演），每条必须有具体测试数据；AI 不确定的分支数据必须继续作为缺口追问，不得凭空臆造

**铁律：**
1. 测试入口必须具体到能直接粘进 bash 跑（不是"跑一下单测"这种泛泛描述）
2. 敏感信息（token、密码、数据库密码）**只写引用**，如 `见 .env 的 DB_PASSWORD`，禁止明文入 PLAN.md
3. 用户没明说轮数 → 默认 **1 轮**（= 所有分支条件覆盖并全部通过一次）
4. 用户在打包问答里 override 为「去掉测试方案」→ 跳过本步和后续测试相关内容，在计划末尾「变更记录」附一行 `用户主动放弃自动化测试`
5. **分支条件清单必须齐全且每条有具体数据** — 不确定的数据必须在打包问答里作为缺口列出，禁止 AI 凭空造值

### 4. 生成 todolist 草案

**先输出到对话，不写文件！**

获取时间：`date "+%Y-%m-%d %H:%M:%S"`

**需求描述规则：** `## 需求描述` 必须含两小节 —— 「用户原话」（原文粘贴，含补充信息按时间追加）+「我的理解（做啥）」（一段话复述目标）。用户贴过的代码/日志/链接/约束汇入 `## 用户资料与引用`，**原文保留，禁止总结重写**，无内容则整节删除。

**测试方案规则：**
- AI 在 3.6 判定为「数据逻辑类」且用户未 override → 计划必须含 `## 测试方案` 章节（放在 `## 技术方案` 之后、`## 文件变更清单` 之前），按 3.6 收集结果填写；Todolist **最后一条永远是「运行自动化测试」原子条目**（见下方模板）
- AI 判定为「直观表现类」或用户 override 去掉测试 → 不写 `## 测试方案` 章节，改为在 `## 技术方案` 之后加一节 `## 测试必要性说明`（一句话说清判定和理由）；todolist 末尾不加测试条目
- **直观表现类改动的 `## 改动前后对比` 必填且给足细节**（文案 diff、ASCII UI 图、字段样例等直观可视化），这是直观类的核心交付物

**change details 规则：**

1. 修改文件标 `path:起始行-结束行`，括号内写 `(现: ...)` 摘要
2. 新建文件不标行号，但列出函数/类/常量名
3. 行号必须用 Read 确认，不得猜测
4. 修改超 30 行，拆成多个子区域

**粒度：** 每条 todo 2-5 分钟可完成，能独立验证，一个功能单元一条。

**格式模板：**

````markdown
# 版本 N: 中文需求名（10-25 字，与目录名同步）

> 创建时间: YYYY-MM-DD HH:mm:ss
> 最后更新: YYYY-MM-DD HH:mm:ss
> 状态: 待执行

## 需求描述

### 用户原话
> 用户最初输入的原始需求，一字不改粘贴（含标点、语气）。后续讨论中补充的关键信息按时间顺序追加在此节。

### 我的理解（做啥）
一句话或一段话复述：这次到底要做什么、解决什么问题、期望效果是什么。用户后续若纠正，更新这段。

## 用户资料与引用

> 承载需求以外的背景资料，支持 todolist 独立执行时参考。无内容则删整节。

- **引用代码 / 数据 / 日志**：用户贴的原文片段，用代码块包住**原文保留，禁止总结重写**
- **外部链接**：API 文档、设计稿、规格说明等 URL
- **业务背景 / 术语 / 约束**：专业术语、兼容要求、历史决策

## 技术方案
选定方案及理由（或单方案思路）。

## 测试必要性说明

> 直观表现类/用户明确放弃测试时出现此节，并整节删除 `## 测试方案`；数据逻辑类删除本节。

一句话：`AI 判定：直观表现类。理由：{具体理由}。本次不含自动化测试，仅以「改动前后对比」呈现效果。`

## 测试方案

> 直观表现类或用户放弃自动化测试则整节删除。

### 入口
```bash
具体测试命令（如 pytest tests/test_xxx.py::test_case -v 或 curl -X POST http://localhost:8080/api/xxx ...）
```

### 数据 / 目标
- 测试对象：xxx（表/接口/函数名）
- 数据来源：xxx（fixture 路径 / 固定测试账号 / 内存构造）

### 环境
- 连接：xxx（敏感值见 .env 的 XXX 字段，PLAN.md 不明文）

### 分支条件清单（必填）
列出本次改动涉及的所有输入分支 / 状态条件 / 边界值，每条必须有**具体测试数据**。**一轮测试必须全部覆盖并通过才算完成**：

- 条件 1：xxx（如 `a == 1` 路径）
  - 数据：输入 `a=1`，预期 xxx
- 条件 2：xxx（如 `a > 1` 路径）
  - 数据：输入 `a=5`，预期 xxx
- 条件 3：xxx（如空值 / 边界）
  - 数据：输入 `a=null`，预期报错 xxx

### 通过标准
- 退出码 0
- 分支条件清单所有条目通过
- （可选）输出字段值 / 正则匹配 / 行数断言

### 轮数
- N 轮（用户指定，默认 1）
- **一轮 = 分支清单全部覆盖并通过一次**

### 失败后处理
- 代码 bug → AI 自动修复相关文件后重跑
- 测试期望不明 → 停下问用户
- 连续 3 次修复失败 → 停下让用户决定方向

## 文件变更清单

**新建:** `path` — 职责
**修改:** `path:行号` — 改什么
**依赖:** 包名 — 用途

## Todolist

- [ ] 1. 任务标题

  **Files:**
  - Create: `path`
  - Modify: `path:42-58`

  ```
  change details:
  新建: path
  - 导出的函数/常量/类型
  修改: path:42-58 (现: 当前内容摘要, 改后: 目标状态)
  - 具体改动点
  ```
----------------------
- [ ] 2. ...
----------------------
- [ ] N. 运行自动化测试（循环直到通过）

  **执行方式：** 按 `## 测试方案.入口` 运行，覆盖 `## 测试方案.分支条件清单` 所有条件，至少完成 `## 测试方案.轮数` 指定的轮数
  **完成定义：** 分支清单所有条件通过，累计达到要求轮数
  **失败处理：** 按 `## 测试方案.失败后处理` 执行（AI 自动分析修复重试；歧义停下问）

## 总结

共 N 条，涉及 X 文件（新建 Y，修改 Z） 。
1. 任务标题 — 做什么、涉及文件、关键函数、改成什么样
2. ...

## 改动前后对比

直观展示宏观效果变化（有数据贴数据，有请求贴请求响应,有画面使用终端画出流程或者画面表示）。

改动前:
（原始现状）

改动后:
（变化后样子）

规则：基于实际代码、数据优先、不重复 todolist。

## 变更记录
- YYYY-MM-DD HH:mm:ss 初始创建
````

## 关键规则

1. **先输出后写** — 草案先展示，用户确认后才落盘
2. **change details 必带行号和当前摘要** — 新建文件列函数/类/常量
3. **零上下文假设** — 写得像给完全不了解项目的人看
4. **原子粒度** — 每条 todo 独立可验证
5. **方案来自代码分析** — 禁止凭空凑数；多方案等用户选
6. **调研先行** — 不读代码不规划
7. **多子系统拆分** — 独立子系统各自一个版本
8. **遵循现有模式，DRY/YAGNI** — 不为假想未来预留
9. **版本已存在拒绝** — 导向 `/xz-update-plan N`
10. **用户原话+资料原文保留** — 需求描述必须含原话和复述理解两段；用户贴的代码/日志/链接进 `## 用户资料与引用`，原文保留不得总结
11. **测试方案按类别决定** — AI 在 3.6 先判定直观表现类/数据逻辑类；数据逻辑类必须含 `## 测试方案` 章节且 todolist 末尾必带测试条目；直观表现类仅保留 `## 测试必要性说明` + `## 改动前后对比`，不写测试方案
12. **分支条件数据必须齐全** — `## 测试方案.分支条件清单` 每条必须有**具体测试数据**（输入值 / 前置状态）；AI 不确定用什么数据时必须**在打包问答里作为缺口列出**，禁止凭空臆造测试值
13. **目录名必须具体** — `N.xxx` 的 `xxx` 含改动对象+核心动作+关键差异，10-25 字；禁空格和 `/\:*?"<>|` 等特殊字符；禁止单用泛化词（优化/重构/修复 等）
14. **一次确认批处理** — 调研完成后，AI 必须把「方案对比 + 选型澄清（3.5）+ 测试类别判定结果 + 3.6 测试信息缺口」一次性打包成一个 AskUserQuestion（或同一段文本）发给用户；禁止拆成多次工具调用连问
15. **能推断不问** — 用户原话、PLAN.md、N-DISCUSS.md、代码现状能推断的字段，直接填默认值并在打包问答里展示 `已填=X（AI 推断）`，用户可选 override
